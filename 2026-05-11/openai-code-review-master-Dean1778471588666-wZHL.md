### 二、需要警惕的风险点

**硬编码的默认 API Token 变更**  
```java
String backendToken = getEnvOrDefault("BACKEND_API_TOKEN", "openai-code-review-internal-token-2026");
```
此处将默认 token 从 `"internal-secret-token"` 改为 `"openai-code-review-internal-token-2026"`。

**风险分析**：
- 虽然实际运行时会优先读取环境变量，但如果开发者未设置环境变量，代码回退到一个**明文硬编码的“有含义”token**，一旦仓库公开或被意外提交，该 token 会直接暴露。
- 旧默认值 `"internal-secret-token"` 虽然也是硬编码，但语义更泛，攻击者可能会尝试但风险相对可控；新值包含“2026”与具体业务标识，更容易被针对性利用。
- 这类设计不符合“安全默认配置”原则：默认值不应提供任何可用的凭证，理想情况应使用类似 `"CHANGE_ME"` 或直接报错/警告退出，强制开发者显式配置。

**架构建议**：
- 去除硬编码的默认 token，改用空字符串，并在获取后发现为空时抛出明确异常或打印严重告警，阻止程序在不安全的状态下启动。
- 敏感凭证永远不能以任何形式出现在代码里，哪怕是作为“示例”或“默认值”。
